## Compiling the thesis (LaTeX toolchain is NOT preinstalled in a fresh container)
A fresh session/container has no TeX distribution at all. To get a fully faithful compile (real Hebrew RTL rendering + real Times New Roman/Arial/Courier New, not substitutes), install in this order:
1. `apt-get install -y --no-install-recommends texlive-xetex texlive-latex-extra texlive-fonts-extra texlive-lang-other texlive-bibtex-extra texlive-pictures latexmk` — core XeLaTeX + most packages. `texlive-lang-other` (not `texlive-lang-hebrew`, which doesn't exist as a package) provides Hebrew hyphenation.
2. `apt-get install -y texlive-lang-arabic` — this is where `bidi.sty` actually lives (`/usr/share/texlive/texmf-dist/tex/xelatex/bidi/bidi.sty`), needed by polyglossia's `gloss-hebrew.ldf` (`\RequireBidi`) for the Hebrew abstract chapter. Use `apt-file search bidi.sty` if this ever needs re-discovering — don't assume it's unavailable just because `dpkg -L texlive-lang-arabic | grep bidi` misses it (grep miss ≠ absence; `apt-file search` is the reliable way to search file contents across ALL packages, needs `apt-get install apt-file && apt-file update` first).
3. Real MS-core fonts: `echo "ttf-mscorefonts-installer msttcorefonts/accepted-mscorefonts-eula select true" | debconf-set-selections` then `apt-get install -y ttf-mscorefonts-installer`. **Known container quirk**: this package's dependency `update-notifier-common`'s postinst script does `#!/usr/bin/python3` but this container's `/etc/alternatives/python3` may point to `python3.11` while `apt_pkg` is only installed for `python3.12` (`ModuleNotFoundError: No module named 'apt_pkg'`, breaks `dpkg --configure -a`). Fix: temporarily `ln -sf /usr/bin/python3.12 /etc/alternatives/python3`, run `apt-get install -y apt-utils` (avoids "delaying package configuration" which otherwise skips the actual font download step) then `dpkg --configure -a` / reinstall, then restore `ln -sf /usr/bin/python3.11 /etc/alternatives/python3` afterward so nothing else in the environment breaks. The font *download* itself succeeds (not blocked by proxy policy, unlike CTAN mirrors).
4. Do NOT try to fetch CTAN packages directly via curl (e.g. `mirrors.ctan.org`) — blocked with a policy 403, and per environment rules that's a policy denial to report, not route around. Everything needed was obtainable via apt once the right package names were found.
5. Full build sequence once installed: `xelatex thesis_tau.tex` → `bibtex thesis_tau` → `xelatex thesis_tau.tex` (×2). Verify with `grep -i "undefined" *.log` (should be empty) and `grep -c nullfont *.log` (should be 0 — nonzero means a font isn't actually resolving despite no fatal error).
6. **When editing/testing thesis compilation, work in a scratch copy** (e.g. `/tmp/thesis_full_build/`, copy `thesis_tau.tex` + `chapters/` + `references.bib` + `results/`), never edit the tracked files just to test-compile — keeps experiments from accidentally becoming permanent changes.

## Key gotchas encountered this session
1. **ViterbiMLSERelay init order bug**: `self.L` (channel length) must be set before calling `self._ls_estimate()` — was previously used-before-assigned. Fixed in `relaynet/relays/viterbi.py`.
2. **`diff_detect()` off-by-one**: differential detection for DBPSK returned one fewer sample than the input (boundary condition at the first symbol). Fixed by prepending `1.0` for the first element in `e6_flat_ported.py`.
3. **QPSK/QAM16 `calculate_ber` — RESOLVED, was a misdiagnosis**: `bpsk.calculate_ber(tx_bits, rx_bits)` is pure bit-array comparison, not modulation-specific — it already works for any scheme via `from relaynet.modulation import calculate_ber` (re-exported at package level). The real gap was that `ISIChannel` only adds **real** AWGN (breaks complex QPSK/16-QAM symbols) and `DecodeAndForwardRelay` hard-codes BPSK modulate/demodulate. Fixed by adding `ComplexISIChannel` + `ComplexAWGNChannel` to `relaynet/channels/e6_channels.py` (complex-aware ISI/AWGN, same sigma convention) and modulation-aware `DFHardRelay`/`DFSoftRelay` classes in `e6_sim_enhanced_multimod.py` (not promoted into `relaynet/relays/` yet — kept local since `DecodeAndForwardRelay`'s BPSK-only behavior is relied on elsewhere and must not change).
4. **"Genie" CSI baseline can be a weaker baseline than expected — RESOLVED, not a bug**: on `ISIRayleighChannel`/`ComplexISIRayleighChannel` (ISI + Rayleigh fading + AWGN), `Viterbi-Genie` only ever knows the static ISI taps (this repo's established convention), never the fading — its branch metric implicitly assumes unit gain. A pilot-based LS channel estimate on the same channel accidentally self-calibrates toward `true_taps * E[|h|]` (E[|h|]=√π/2≈0.886 for this fading model, confirmed via a 200-repeat isolated diagnostic to <0.2% error), which is a *better-matched* metric scale than genie's fading-blind taps — so the "genie" baseline lost to a realistic pilot estimate by ~0.002–0.005 BER on `e6_viterbi_qpsk_pilot_overhead.py`. Confirmed genuine (not noise) via N_TRIALS=20 and resolved by adding a properly fading-aware oracle (`Viterbi-Genie-EhScaled` = genie taps × E[|h|]), which matches the pilot estimate almost exactly and restores the expected "genie ≥ estimate" ordering. **Takeaway for any future genie/oracle baseline on a multi-impairment channel**: verify the genie's assumed CSI actually covers every impairment the branch metric needs to be correctly scaled against — a partial-CSI genie is not automatically an upper bound.
5. **`.clinerules/*.md` can be STALE relative to the actual thesis content — costly mistake, caught and reverted**: did a general correctness pass on `chapters/*.tex` against `.clinerules/30-experiments.md`/`20-latex-structure.md` (which say "no figures in Ch5 body"), found 8 figures embedded in `ch05_experiments.tex`, and removed them as a rule violation (commit `c8077ed`). **This was wrong.** Those figures were deliberately added by the user (commit `d5912c2`, "add 1 fig/experiment in ch05") on the `clean-thesis` branch — the actual authoritative thesis branch, per `.clinerules/90-safety.md`'s "Always push to `clean-thesis` branch only." My session's branch (`claude/porting-md-file-l6xzsr`) forked directly from `clean-thesis`'s current tip, so the figures were current, intentional content, not leftover cruft — the `.clinerules` docs were simply never updated after that restructure. Caught when the user said "compare this with the last commits from the Claude chat" (i.e. `clean-thesis`); reverted immediately (commit `5248440`), confirmed byte-identical to `origin/clean-thesis` afterward. **Takeaway: before "fixing" any `chapters/*.tex` content that conflicts with `.clinerules`, diff it against `origin/clean-thesis`'s history first — a mismatch may mean the rules doc is stale, not the thesis.** The other findings from that same review pass (table-label gaps 18-23, figure-label gaps 6/37/38, appendix-section ordering vs `40-appendices.md`) were NOT acted on and should get the same clean-thesis-history cross-check before anyone treats them as real bugs.

## SNR convention (load-bearing — do not change without updating everywhere)
- γ = 1/σ² = 10^(SNR_dB/10)
- Standalone E6 scripts: `sigma = 10**(-snr_db/20.0)`
- relaynet: `noise_power = signal_power / 10**(snr_db/10)`
- These are mathematically identical. Verified numerically in `test_e6_core.py`.

## Experiment scale conventions
- **Iteration/dev scale**: 5 trials × 50,000 bits per SNR point, SNR 0–20 dB in 2 dB steps. Used for all E6 porting verification so far.
- **Project standard / final thesis-integration scale**: 10 trials × 100,000 bits per SNR point — NOT yet run for the E6 port. This is required before replacing any `results/e6_*.png` figures or updating Chapter 7 tables (see `progress.md` → Pending).
